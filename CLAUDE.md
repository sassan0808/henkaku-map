# 変革手段の地図 - CLAUDE.md

## プロジェクト概要

変革手段を体系化したWebサイト。個人で育てて、筋がよければLotsful用にブランチする。

## 技術構成

- **Docusaurus** (タグシステム Lv3)
- **GitHub Pages** で公開（リポジトリ: sassan0808/henkaku-map）
- mdのfrontmatterでタグ管理

## 設計思想

- 5つの変革の柱を軸に体系化（課題→処方箋の構造）
- 階層はツリーではなく**グラフ構造**
  - 同じ概念が複数の文脈に横断する（例：副業・フリーランスは「人・能力」「実行・プロセス」両方に関係）
- 1ページに複数文脈のセクションを持つ
- タグで横断的に辿れるようにする
- 階層は後から変えられる前提で設計

## 5つの変革の柱

| 柱 | 課題 | 手段 |
|---|---|---|
| 知見・意思決定の変革 | 何をすべきか判断できない | スポットコンサル、市場調査 |
| 人・能力の変革 | 人の質・意識を変えたい | 社内研修、越境学習 |
| 体制・人材の変革 | 必要なリソースが社内にいない | 人材登用、副業・フリーランス、顧問 |
| 実行・プロセスの変革 | 実行が伴わない・効率が悪い | ツール導入、BPR、PMO |
| 販路・外部資本の変革 | 自社単独の成長に限界がある | 販路開拓、アライアンス、M&A |

## 進化方針（2026-04-18 決定）

### サイトの役割分担

henkaku-map は「静的な情報場」として育てる。動的な診断・出力は LLM ツール側に任せる。

```
[henkaku-map サイト（公開）]     [LLM ツール（別途・商談用）]
Phase 1: 症状から探す       →   議事録 + henkaku の内容を input
Phase 2: ペイオフマトリクス  →   診断結果 + 推奨手段 + Why Now? を output
（静的・Docusaurus）              （Claude Project or カスタムGPT）
```

### Phase 1〜2：サイト側でやること

- **Phase 1：** 症状タグの追加（`症状_離職`, `症状_意思決定遅延` など）＋症状インデックスページ
- **Phase 2：** 各手段に `difficulty` / `impact` を frontmatter で定義し、ペイオフマトリクスとして可視化

### Phase 3〜4：LLM ツール側でやること

- 商談後に議事録を貼ると、推奨手段・優先順位・Why Now? が出力される
- henkaku の全 MD をシステムプロンプトに埋め込む（現状のコンテンツ量なら余裕で収まる）
- 最初は Claude Project の Custom Instructions として実装、必要に応じて RAG 化

### 判断の背景

- 診断フォームより「議事録を貼る」ほうが商談フローに近い
- Docusaurus に複雑な状態管理を持ち込まない（壊さない）
- コンテンツ（情報場）の質を上げることがサイト側の本質的な役割

## リンク設計の方針：循環構造で説得力を高める

このサイトのコンテンツは「一方向の参照」ではなく、**層をまたいだ循環構造**として設計する。

### 3層構造とリンクの流れ

```
課題マップ（症状・問題）
  ↕ 双方向リンク
変革マップ（手段・打ち手）
  ↕ 双方向リンク
フレームワーク・参考書（理論的根拠）
```

各層は独立したページ群だが、相互にリンクを張ることで「なぜそれをやるべきか」の根拠が循環し、読者がどこから入っても全体像に到達できる構造を作る。

### 具体的なリンクのつけ方

| ページの種類 | リンクの方向 | 目的 |
|------------|------------|------|
| 課題マップ | → 変革マップ（手段） | 課題から打ち手へ誘導 |
| 変革マップ | → 課題マップ（どの課題に効くか） | 手段の適用範囲を明示 |
| 変革マップ | → フレームワーク・参考書（理論根拠） | 「なぜ有効か」の説得力を付加 |
| フレームワーク・参考書 | → 変革マップ（具体的な打ち手） | 理論から実践へ橋渡し |
| フレームワーク・参考書 | → 課題マップ（どの症状に効くか） | 理論の適用場面を具体化 |

### 設計のポイントに「参考フレーム」を入れる

変革マップの各ページにある「設計のポイント」や「注意点」のセクションには、
その主張を裏付けるフレームワーク・書籍への引用を積極的に加える。

```md
> **参考フレーム：** [OKR](/docs/フレームワーク・参考書/フレームワーク/OKR) の考え方が有効。
> OKRは「何に挑戦したか」を評価する構造であり、プロジェクト参加との親和性が高い。
```

これにより「経験則ではなく、理論的に正しい打ち手」として読者に届く。

### 循環構造の目指す状態

- 症状チェックリスト → 変革手段 → フレームワーク → 書籍要約 → また別の変革手段
- という回遊が自然に生まれ、サイト全体が「組織変革の知識ベース」として機能する
- LLMツールにこのMDを渡したときも、リンクの文脈ごと渡せるため推論精度が上がる

## フレームワーク・書籍を追加するときのチェックリスト

新規ページを作ったら以下を必ず確認する：

1. **課題マップへの逆リンク** — 真因の仮説レベルで「なぜこの問題が起きるか」を説明できる症状に引用する
2. **変革マップ手段ページへのリンク** — 「参考フレーム」として引用できる手段ページに追記する
3. **変革マップ概要ページの「関連する理論・手法」** — 該当する柱の概要ページにリンクを追加する
4. **フレームワーク・参考書/概要.md の一覧表** — 種別・一言説明・関連変革テーマを追記する
5. **既存フレームワークとの相互リンク** — 比較・補完関係にある既存ページに双方向リンクを貼る

## 設計原則

1. 完璧な構造を最初に作らない
2. 書きながら構造を発見する
3. タグさえ設計すれば階層移動のコストは低い
4. Claude Codeへの口頭指示で編集・拡張していく
5. 新しいページを作るときは「どの層と双方向リンクを貼るか」を必ず考える
