# 変革手段の地図 - CLAUDE.md

## プロジェクト概要

変革手段を体系化したWebサイト。個人で育てて、筋がよければLotsful用にブランチする。

## 技術構成

- **Docusaurus** (タグシステム Lv3)
- **GitHub Pages** で公開（リポジトリ: sassan0808/henkaku-map）
- mdのfrontmatterでタグ管理

## 設計思想

- 5つの変革の柱を軸に体系化（課題→処方箋の構造）
- 階層はツリーではなく**グラフ構造**
  - 同じ概念が複数の文脈に横断する（例：副業・フリーランスは「人・能力」「実行・プロセス」両方に関係）
- 1ページに複数文脈のセクションを持つ
- タグで横断的に辿れるようにする
- 階層は後から変えられる前提で設計

## 5つの変革の柱

| 柱 | 課題 | 手段 |
|---|---|---|
| 知見・意思決定の変革 | 何をすべきか判断できない | スポットコンサル、市場調査 |
| 人・能力の変革 | 人の質・意識を変えたい | 社内研修、越境学習 |
| 体制・人材の変革 | 必要なリソースが社内にいない | 人材登用、副業・フリーランス、顧問 |
| 実行・プロセスの変革 | 実行が伴わない・効率が悪い | ツール導入、BPR、PMO |
| 販路・外部資本の変革 | 自社単独の成長に限界がある | 販路開拓、アライアンス、M&A |

## 進化方針（2026-04-18 決定）

### サイトの役割分担

henkaku-map は「静的な情報場」として育てる。動的な診断・出力は LLM ツール側に任せる。

```
[henkaku-map サイト（公開）]     [LLM ツール（別途・商談用）]
Phase 1: 症状から探す       →   議事録 + henkaku の内容を input
Phase 2: ペイオフマトリクス  →   診断結果 + 推奨手段 + Why Now? を output
（静的・Docusaurus）              （Claude Project or カスタムGPT）
```

### Phase 1〜2：サイト側でやること

- **Phase 1：** 症状タグの追加（`症状_離職`, `症状_意思決定遅延` など）＋症状インデックスページ
- **Phase 2：** 各手段に `difficulty` / `impact` を frontmatter で定義し、ペイオフマトリクスとして可視化

### Phase 3〜4：LLM ツール側でやること

- 商談後に議事録を貼ると、推奨手段・優先順位・Why Now? が出力される
- henkaku の全 MD をシステムプロンプトに埋め込む（現状のコンテンツ量なら余裕で収まる）
- 最初は Claude Project の Custom Instructions として実装、必要に応じて RAG 化

### 判断の背景

- 診断フォームより「議事録を貼る」ほうが商談フローに近い
- Docusaurus に複雑な状態管理を持ち込まない（壊さない）
- コンテンツ（情報場）の質を上げることがサイト側の本質的な役割

## 設計原則

1. 完璧な構造を最初に作らない
2. 書きながら構造を発見する
3. タグさえ設計すれば階層移動のコストは低い
4. Claude Codeへの口頭指示で編集・拡張していく
