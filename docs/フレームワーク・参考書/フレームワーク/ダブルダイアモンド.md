## 一言で言うと

> 「解決策を考える前に、本当に解くべき問題を定義する」。このプロセスを2つのダイアモンド（発散→収束）で表した、デザイン思考の基本構造。

---

## 概要

ダブルダイアモンドは、英国デザインカウンシルが2005年に提唱したデザインプロセスモデル。

```
     発散              収束          発散              収束
      ◇────────────────◇            ◇────────────────◇
     /                  \          /                  \
    /   Discover          \  Define/   Develop          \ Deliver
   /  （現実を広く見る）    \  ↓   /  （解決策を広く探る） \  ↓
  ●                        ●────●                        ●
 スタート              正しい問いの定義              解決策の実装
```

**第1ダイアモンド：「正しい問いを立てる」（問題空間）**
- Discover（発見）：現場・ユーザー・データから現実を広く観察する
- Define（定義）：観察を統合し、本当に解くべき問題を一文で定義する

**第2ダイアモンド：「正しく解く」（解決空間）**
- Develop（展開）：解決策のアイデアを広く発散させ、プロトタイプを作る
- Deliver（実装）：テスト・改善を繰り返し、実際に届けられる形にする

---

## なぜ「2つ」のダイアモンドが必要か

ほとんどの組織が失敗するのは、**第1ダイアモンドをスキップして第2ダイアモンドに飛び込む**からだ。

```
よくある失敗パターン

  「売上が落ちている」（症状の認識）
    ↓ 問題を深く調べない
  「じゃあ新しい営業ツールを導入しよう」（解決策に飛びつく）
    ↓
  ツールを導入したが売上は改善しない
    ↓
  「問題の定義が間違っていた」（本当の原因は別にあった）

正しいプロセス（ダブルダイアモンド）

  「売上が落ちている」（症状の認識）
    ↓ 第1ダイアモンド：観察・インタビュー・データ分析
  「既存顧客の解約が増えている。理由は価値実感の不足」（本当の問い）
    ↓ 第2ダイアモンド：解決策の探索と実装
  「カスタマーサクセス体制の強化」（正しい打ち手）
```

---

## 4フェーズの詳細

### 1. Discover（発見）— 現実を広く観察する

**やること：**
- 現場・顧客・ユーザーへのインタビュー
- データの観察（数字の背景を読む）
- 競合・市場・業界トレンドのリサーチ
- 「思い込み」を一度脇に置く

**重要な姿勢：** 「解決策を考えない」。この段階では答えを出さず、問いを増やす。

### 2. Define（定義）— 本当に解くべき問いを一文にする

**やること：**
- 観察から得た情報を統合・パターン化
- 「問いの絞り込み」：何が本当の問題か
- HMW（How Might We）形式で問いを立てる

**HMW形式の例：**
- ✗「売上を増やすにはどうすればいいか」（漠然としすぎ）
- ○「既存顧客が価値を実感できないのはなぜか、それをどう解消するか」

> **この定義が間違っていると、第2ダイアモンドで何をやっても無駄になる。**

### 3. Develop（展開）— 解決策を広く探索する

**やること：**
- アイデアの発散（ブレスト・類似事例の収集）
- プロトタイプの作成（MVPでも可）
- 仮説の明示：「これをやれば問いが解決されるはず」

リーンスタートアップの「Build」フェーズと重なる（[リーンスタートアップ](/docs/フレームワーク・参考書/書籍要約/リーンスタートアップ) 参照）。

### 4. Deliver（実装）— テストして届ける

**やること：**
- ユーザーテスト・パイロット実施
- フィードバックをもとに改善
- スケールアップ or ピボットの判断

## このサイト（変革手段の地図）自体がダブルダイアモンド構造

実はこのサイト全体が、ダブルダイアモンドの構造で設計されている。

```
第1ダイアモンド（問題空間）
  ↓ Discover：組織の症状・課題を広く観察する
  課題マップ（A〜G、40以上の症状）
  ↓ Define：症状チェックリストで「自社の本当の問い」を定義する
  症状チェックリスト

第2ダイアモンド（解決空間）
  ↓ Develop：変革手段の選択肢を広く探索する
  変革マップ（7つの変革領域）
  ↓ Deliver：理論的根拠をもとに実装する
  フレームワーク・参考書
```

組織変革の支援においても、「打ち手の提案」より先に「問いの定義」に時間をかけることが、成功確率を大きく高める。

## AI時代でダブルダイアモンドが重要になる理由

AI は「解く力」を劇的に高めるが、**「何を解くか」の問いは与えられない。**

```
AI時代の構造的リスク

  AIが解く力を高める
    ↓
  問題定義なしに「とりあえずAIでやってみよう」が加速
    ↓
  間違った問いを高速で解く → 間違った答えが大量生産される
    ↓
  「AIを導入したが成果が出ない」

AI時代でのダブルダイアモンドの役割

  第1ダイアモンド（問いを立てる）：人間の仕事
    → AIは観察・分析を支援できるが、「何が本質的な問いか」は人間が判断する
  第2ダイアモンド（解く）：AIが最大限に力を発揮する領域
    → 解決策の探索・プロトタイプ・テストをAIが高速化する
```

**「問いを立てる力」こそが、AI時代に人間に求められる最重要スキルになる。**

---

## 組織変革への応用

| フェーズ | 変革プロジェクトでの実践 |
|---------|----------------------|
| Discover | 現場インタビュー・データ観察・他社事例収集 |
| Define | 経営課題を「HMW形式」の問いに変換する |
| Develop | 変革手段の候補をリストアップし、小さく試す |
| Deliver | パイロット → 効果測定 → 全社展開 |

**第1ダイアモンドへの投資が足りない組織の典型症状：**
- 「変革プロジェクトが途中で止まる」
- 「施策をやったが手応えがない」
- 「何から手をつければいいかわからない」

これらはすべて、問いの定義が曖昧なまま解決策に飛びついた結果だ。
