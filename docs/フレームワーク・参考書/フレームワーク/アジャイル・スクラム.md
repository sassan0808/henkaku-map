## アジャイルとは何か

**アジャイル（Agile）** は、2001年に17人のソフトウェア開発者が「アジャイルソフトウェア開発宣言」としてまとめた開発思想。
もとはITの世界で生まれたが、現在は組織運営・事業開発全般に応用されている。

### アジャイル宣言の4つの価値

| 〜より | こちらを重視する |
|--------|--------------|
| プロセスやツール | **個人と対話** |
| 包括的なドキュメント | **動くソフトウェア（成果物）** |
| 契約交渉 | **顧客との協調** |
| 計画に従うこと | **変化への対応** |

右側を重視するというのは、左側を無視するということではない。**変化が速い環境では右側の優先度が高い**、という主張だ。

## スクラムとは何か

**スクラム（Scrum）** は、アジャイルを実践するための具体的なフレームワーク。
ラグビーのスクラムのように「チームが一体となって前進する」イメージから名付けられた。

### スクラムの3つの役割

| 役割 | 責任 |
|------|------|
| **プロダクトオーナー（PO）** | 何を作るか・優先順位を決める。顧客・事業の代表 |
| **スクラムマスター** | チームがスムーズに動けるよう障害を取り除く。プロセスの番人 |
| **開発チーム** | 実際に作る。自律的・クロスファンクショナルなチーム |

### スプリント（繰り返しサイクル）

スクラムは **スプリント** と呼ばれる1〜4週間の短いサイクルを繰り返す。

```
スプリント計画（何をやるか決める）
  ↓
スプリント実行（1〜4週間）
  ↓
スプリントレビュー（成果を確認・顧客フィードバック）
  ↓
スプリントレトロスペクティブ（やり方を振り返る）
  ↓
次のスプリント計画...
```

## なぜアジャイルが「変革」に必要か

### ウォーターフォール型の限界

従来の「ウォーターフォール型」プロジェクト管理は、変革業務に向かない。

```
ウォーターフォール：
要件定義 → 設計 → 開発 → テスト → リリース（6〜12ヶ月後）
  → リリース時に「こんなものが欲しかったわけではない」が頻発
  → 環境が変わってニーズが変わっている
```

変革プロジェクトは「正解が最初からわかっている」種類のプロジェクトではない。
だからこそ、動きながら学ぶアジャイルが適している。

### アジャイルが変革にもたらすもの

| 変革の課題 | アジャイルのアプローチ |
|-----------|-------------------|
| 何が正解かわからない | 小さく試して、動いた結果から学ぶ |
| 計画が陳腐化する | 1〜2週間のサイクルで計画を更新し続ける |
| チームの合意が取れない | スプリントレビューで全員が成果を見て議論する |
| 進捗が見えない | 毎スプリント動くアウトプットが出る |

## 「組織運営のアジャイル化」という応用

IT開発を超えて、組織そのものをアジャイルに設計する動きが広がっている。

### アジャイル組織の特徴

- **小さな自律チーム** — 大きな部門ではなく、スクワッドやフィーチャーチームが意思決定の単位
- **明確なオーナーシップ** — 誰が何に責任を持つかが明確（プロダクトオーナー的な役割）
- **高速フィードバックループ** — 顧客・現場の声を素早く意思決定に反映する仕組み
- **心理的安全性** — 失敗から学ぶことを奨励する文化

### 日本企業でアジャイルが難しい理由

- 評価制度が個人の成果ではなくチームの成果を評価しにくい
- 「スクラムマスター」のような役割が評価されない
- 上長の承認なしに動けないカルチャー
- 失敗を学習として扱わず責任追及する文化

これらはすべて、変革マップの課題と重なる。

---

## 実務での使い方

- **変革プロジェクトの運営** — 月次進捗報告ではなく、週次スプリントで動く
- **新規事業開発** — 完璧な事業計画より、最小のプロトタイプを早く作って検証する
- **DX推進** — 全社一斉展開より、1部門・1業務でスプリントを回してから横展開
