## PDCAとは何か

**PDCA（Plan → Do → Check → Act）** は、製造業の品質管理から生まれた改善サイクルだ。
1950年代にデミングが提唱し、日本の高度成長期に広く普及した。

```
Plan（計画）→ Do（実行）→ Check（評価）→ Act（改善）→ Plan...
```

**PDCAが機能する前提条件：**
- 環境が比較的安定している
- 計画段階で必要な情報がそろっている
- 正解が事前に予測できる

高度成長期の製造業では、これらの条件が成立していた。

---

## なぜPDCAが機能しなくなったか

現代のビジネス環境ではPDCAの前提が崩れている。

| 前提 | 高度成長期 | 現在 |
|------|-----------|------|
| 環境の変化速度 | 遅い（年単位） | 速い（月・週単位） |
| 計画の有効期間 | 長い | 短い（すぐ陳腐化） |
| 競合の動き | 予測可能 | 予測困難 |
| 正解の有無 | 正解がある | 正解が事前にわからない |

**PDCAの最大の問題：**「Plan（計画）」に時間をかけすぎる。計画が完成するころには、前提が変わっている。

---

## OODAループとは何か

**OODA（Observe → Orient → Decide → Act）** は、米軍のジョン・ボイド大佐が提唱した意思決定フレームワーク。
空中戦での素早い判断のために設計されたが、ビジネスの不確実性対応に応用されている。

```
Observe（観察）→ Orient（状況判断）→ Decide（意思決定）→ Act（行動）→ Observe...
```

| フェーズ | 内容 |
|---------|------|
| **Observe（観察）** | 現実の状況をありのままに把握する。データ・現場・顧客の声を集める |
| **Orient（方向付け）** | 観察した情報を解釈し、「今何が起きているか」のモデルを作る。ここが最重要 |
| **Decide（決断）** | 仮説に基づいて最善と思われる行動を選ぶ |
| **Act（行動）** | 素早く実行し、結果を観察にフィードバックする |

---

## PDCAとOODAの本質的な違い

| 観点 | PDCA | OODA |
|------|------|------|
| 起点 | 計画（Plan）から始まる | 観察（Observe）から始まる |
| 前提 | 正解がある程度見えている | 正解は動きながら発見する |
| スピード | 1サイクルが長い | 高速に何度も回す |
| 失敗の扱い | 計画からの逸脱（悪） | 学習のための情報（善） |
| 向く環境 | 安定・予測可能 | 変化が速い・不確実 |

---

## 組織変革との接続

### なぜ変革プロジェクトはPDCAで失敗するか

変革プロジェクトに「完璧な計画を作ってから動く」PDCA的思考を適用すると、

1. 計画策定に数ヶ月かかる
2. 計画中に環境が変わる
3. 完成した計画がすでに古い
4. それでも計画通りに動こうとする
5. 現実とのズレが広がり、プロジェクトが機能不全に

### OODAで変革プロジェクトを設計すると

```
まず小さく動く（Act）
  → 現場の反応を観察する（Observe）
  → 何が起きているかを解釈する（Orient）
  → 次のアクションを決める（Decide）
  → また動く（Act）
```

完璧な計画より「動いて学ぶ」サイクルを速く回すことが変革の推進力になる。

### Orient（方向付け）が最も重要な理由

OODAで見落とされがちだが、最重要フェーズは **Orient** だ。
同じ情報を見ても、解釈が違えば行動が変わる。

**Orient の質を高めるもの：**
- 多様な経験・視点を持つメンバー（→ 越境学習、外部人材の活用）
- 心理的安全性（異論を言える文化）
- データに基づく現状把握（→ データ基盤整備）

---

## 実務での使い方

- **新規事業・新市場参入** — 正解が見えないので、小さなPoCをOODA的に回す
- **変革プロジェクト** — 月次計画より週次の観察→判断→行動サイクルを設計する
- **AI活用推進** — ツールを入れて観察→改善というOODAサイクルが最適
