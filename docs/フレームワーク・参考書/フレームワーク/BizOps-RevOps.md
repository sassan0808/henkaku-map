## 一言で言うと

> 営業・マーケ・CS・経営企画などの「縦割り部門」をデータでつなぎ、全体最適の意思決定と実行を実現する組織・業務の設計思想。

---

## BizOpsとは

**BizOps（Business Operations）** は、事業戦略と現場オペレーションの間を橋渡しする機能・思想です。

シリコンバレーのテック企業（Google、Meta、Airbnbなど）が2010年代に確立し、「経営判断のスピードを上げる」ために生まれました。

### BizOpsが解く問題

| よくある状態 | BizOpsが目指す状態 |
|------------|-----------------|
| 経営と現場で見ている数字が違う | 全社で単一のデータソースを共有 |
| 意思決定に時間がかかる（データが取れない） | リアルタイムに近いデータで即断できる |
| 戦略が現場に落ちない | 戦略→KPI→施策→結果を一気通貫で管理 |
| 部門間でデータが分断されている | データが横断的に流れる構造 |

### BizOpsの主な役割

1. **データの一元化** — 部門バラバラのデータを統合、単一の真実（Single Source of Truth）を作る
2. **KPI設計と可視化** — 事業の健全性を測る指標を設計し、全社で共有する
3. **意思決定支援** — 経営陣・事業部が判断するためのデータ・分析を即座に提供
4. **オペレーション改善** — ボトルネックを特定し、プロセスを改善する

## RevOpsとは

**RevOps（Revenue Operations）** は BizOps の考え方を「収益プロセス」に特化させたものです。

**マーケティング・営業・カスタマーサクセスの3部門をデータで統合し、収益を最大化します。**

### なぜ3部門の統合が必要か

```
従来のサイロ構造
  マーケティング → 「リード数」を追う
  営業          → 「受注数・売上」を追う
  CS            → 「継続率・満足度」を追う
  ↓
  引き継ぎで情報が落ちる
  どこにボトルネックがあるか見えない
  顧客体験がバラバラになる

RevOpsの統合構造
  マーケ → 営業 → CS を一本のパイプラインとして管理
  ↓
  どのステップで失注・解約が起きているかが見える
  顧客LTV（生涯価値）を軸に全体最適できる
```

### RevOpsが管理する指標の例

- リードから受注までの転換率（各ステップ別）
- 平均受注期間（リードタイム）
- 顧客獲得コスト（CAC）と顧客生涯価値（LTV）のバランス
- チャーン率（解約率）とNPS

## AI時代でBizOps・RevOpsが重要になる理由

AIは「データがあるところ」にしか力を発揮できません。

```
AIの活用レベル
  Lv1：特定業務の自動化（データ不要）
  Lv2：分析・予測（部門単位のデータが必要）
  Lv3：全社最適・意思決定支援（横断データが必要）← ここがAI本来の価値
  Lv4：自律的な事業運営（リアルタイム統合データが必要）
```

**Lv3以上のAI活用は、BizOps・RevOpsの基盤なしには実現できません。**

逆に言えば、AIを「部分的な自動化」で終わらせている企業の多くは、データ統合ができていないことが根本原因です。

### 具体的な活用シナリオ

| シナリオ | 必要なデータ統合 |
|---------|--------------|
| 「なぜ今月の受注が落ちたか」をAIが自動分析 | マーケ・営業・外部市場データの統合 |
| 解約リスクの高い顧客をAIが事前検知 | 営業・CS・プロダクト利用データの統合 |
| 最適な価格・提案をAIがリアルタイム生成 | 顧客履歴・競合・コストデータの統合 |
| 経営ダッシュボードをAIが自動更新・コメント | 全社KPIデータの統合 |

## 日本企業が取り組む際の現実的なステップ

### Step 1：データの棚卸し（まず現状把握）
- 部門ごとに何のデータをどこに持っているかを可視化
- SFA・MA・CSツール・ERPなどのデータがつながっているか確認

### Step 2：Single Source of Truthの構築
- データウェアハウス（BQ・Snowflakeなど）or 統合ダッシュボードの整備
- 「どの数字が正しいか」で揉めない状態を作る

### Step 3：KPIの統一設計
- 部門横断で合意できるKPIツリーを設計
- OKRとの連動設計（[OKR](/docs/フレームワーク・参考書/フレームワーク/OKR) 参照）

### Step 4：RevOps/BizOps担当の設置
- 最初は兼務・小チームでも可
- 「データを見て動く文化」を作ることが目的
