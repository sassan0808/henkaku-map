## 一言で言うと

> チームの設計こそが、組織のアウトプット速度を決める。「何をどう作るか」より「誰がどう協力するか」の構造を先に設計せよ。

---

## 出発点：コンウェイの法則

> **「システムを設計する組織は、その組織のコミュニケーション構造をコピーしたシステムを生み出す」**
> — メルヴィン・コンウェイ（1967年）

つまり、**組織の構造がそのままプロダクト・サービスの構造になる**。

よくある失敗：
- 組織を変えずにマイクロサービス化しようとする → 巨大なモノリスが残る
- 縦割り組織のままアジャイルを導入する → 「アジャイルごっこ」になる
- 部門間の壁があるまま横断プロジェクトを立ち上げる → 承認フローで止まる

### 逆コンウェイ戦略（Inverse Conway Maneuver）

「システムをどう設計したいか」から逆算して「組織をどう設計するか」を決める。
アーキテクチャの変革は、チームの変革から始める。

---

## 4つのチームタイプ

すべてのチームをこの4つに分類する。

### 1. ストリームアラインドチーム（Stream-aligned Team）

**価値の流れ（ビジネスドメイン・顧客）に沿って編成された主力チーム。**

- 組織の大部分はこのタイプが担う
- 顧客への価値提供を継続的に行う
- 他チームへの依存を最小化し、自律的に動く
- 例：「ECサイトのカート機能チーム」「法人営業支援チーム」

**日本でよくある問題：** 機能部門（開発、QA、デザイン、営業）に縦割りで人が置かれ、ストリームアラインドチームが存在しない。

### 2. イネーブリングチーム（Enabling Team）

**他チームの能力を引き上げることに特化したチーム。**

- コンサル・コーチ的な役割
- 直接プロダクトを作らず、他チームが自律できるよう支援する
- 新しい技術・手法・知識を社内に普及させる
- 依存関係を作らず、サポートが終わったら離れる
- 例：「アジャイルコーチチーム」「データエンジニアリング支援チーム」

### 3. コンプリケイテッドサブシステムチーム（Complicated Subsystem Team）

**高度な専門性が必要な領域を担当するチーム。**

- 他チームが扱うには複雑すぎる領域を切り出す
- 少数の深い専門家で構成
- 例：「決済エンジン専門チーム」「機械学習基盤チーム」

### 4. プラットフォームチーム（Platform Team）

**他チームが自律的に動けるよう、共通基盤を提供するチーム。**

- インフラ・共通ツール・APIを「サービス」として提供
- 他チームの認知負荷を下げることが目的
- 例：「社内データ基盤チーム」「認証・権限管理チーム」

## 3つのインタラクションモード

チーム間の協力の仕方も、3つのパターンに絞る。

| モード | 内容 | 期間 |
|--------|------|------|
| **コラボレーション** | 2チームが密接に協力して新しい何かを探索する | 短期・集中的 |
| **X-as-a-Service** | 片方がAPIやサービスを提供し、もう片方が使う | 継続的・独立的 |
| **ファシリテーション** | イネーブリングチームが他チームを支援する | 期限付き・移行期 |

**重要：** コラボレーションは密度が高い分、コストも高い。ずっと続けると疲弊する。探索フェーズだけに使い、安定したら X-as-a-Service に移行するのが理想。

## 認知負荷（Cognitive Load）という概念

チームに与えるべき仕事の範囲を考えるとき、**認知負荷**が設計の基準になる。

人間のチームが同時に扱える複雑さには限界がある。この限界を超えると：
- 品質が落ちる
- 意思決定が遅くなる
- メンバーが消耗する

**チームの設計 = 認知負荷を適切に分割する設計**

| 認知負荷の種類 | 内容 | 対策 |
|-------------|------|------|
| 本質的な複雑さ（やるべきこと） | ドメイン知識・ビジネスロジック | チームのサイズ・専門化で対応 |
| 付随的な複雑さ（やらされること） | インフラ運用・ツール管理など | プラットフォームチームに委譲 |
| 外来的な複雑さ（余計なこと） | 不要な依存関係・割り込みなど | チーム間設計で排除 |

## 日本企業への示唆

**なぜ「新規事業部を作っても機能しないか」の構造的説明：**

```
典型的な失敗パターン
  新規事業部を設置
  ↓
  でも既存の機能部門（開発・法務・財務）への依存が残る
  ↓
  承認・リソース確保のたびに既存部門を通る
  ↓
  既存事業の優先度に飲み込まれる
  ↓
  スピードが出ない → 成果が出ない → 解散

チームトポロジー的な解決
  新規事業チームをストリームアラインドチームとして設計
  ↓
  インフラ・法務・財務は「プラットフォームチーム」としてサービス化
  ↓
  新規事業チームが自律的に動ける構造を作る
```

## このサイトの文脈での応用

```
チームトポロジー
  ↓
プロジェクト型組織設計（ストリームアラインドチームの設計）
タレントマネジメント（チームに必要な認知負荷を設計する）
両利きの経営（探索チームをストリームアラインドで設計する）
```

「変革チームをどう設計するか」という問いに対して、チームトポロジーは具体的な答えを提供する。「誰かが変革をやるべき」という曖昧な状態から、「どのチームタイプで、どのインタラクションモードで動かすか」という設計の議論に移行できる。
