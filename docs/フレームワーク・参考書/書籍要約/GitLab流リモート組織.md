- **著者：** 千田和央
- **刊行：** 2023年
- **副題：** ドキュメントの活用でオフィスなしでも最大の成果を出すグローバル企業のしくみ

---

## 一言で言うと

> リモート組織の設計原則は、そのままAIレディな組織の設計原則である。「ドキュメントに書く」「非同期で動く」「透明性をデフォルトにする」——この3つを実践した組織だけが、AIを本当の武器にできる。

---

## 「リモート組織」という言葉の誤解

この本のタイトルに「リモート」とあるが、**出社しているかどうかは本質ではない**。

GitLabが実践しているのは「場所に依存しない組織設計」ではなく、**「人間の記憶や口頭確認に依存しない組織設計」**だ。

この発想は、出社前提の組織にもそのまま適用できる。むしろ、AI時代に生き残る組織の条件として読むべき内容である。

---

## GitLabが実践する5つの設計原則

### 原則1：ドキュメント・ファースト

**「書かれていないことは、存在しないのと同じ」**

会議で決まったことも、誰かが持っている知識も、ドキュメントに書かれていなければ組織の資産にならない。
GitLabでは全社ルール・プロセス・意思決定の背景が、誰でも参照できるハンドブックに集約されている。

```
口頭で話す
  → 記憶に残るが、伝達されない
  → 人が変わると消える
  → 再現性がない

ドキュメントに書く
  → いつでも参照できる
  → 人が変わっても残る
  → AI が学習・処理できる ← AI時代の核心
```

**AI時代との接続：**
AIはテキスト（ドキュメント）を処理する。ドキュメント文化が根付いた組織は、AIに「社内知識」を渡せる。
ドキュメントのない組織では、AIに渡せるコンテキストがなく、汎用的な回答しか得られない。

---

### 原則2：非同期コミュニケーション・デフォルト

**「即座の返答を前提とした仕事の仕方」を疑う**

リアルタイムの会議・チャットは「その瞬間に全員が揃っていること」を前提にしている。これは時差・場所だけでなく、**深い集中仕事との両立**においても障壁になる。

GitLabでは意思決定・議論をドキュメントで非同期に行い、会議は「同期が必要な目的があるときだけ」に絞る。

| 同期コミュニケーション | 非同期コミュニケーション |
|---------------------|----------------------|
| 会議、即時チャット | ドキュメント、コメント、Issue |
| 全員が同じ時間を使う | 各自のペースで処理できる |
| 記録が残りにくい | 議論の経緯が自動的に記録される |
| 集中を中断させる | 集中時間を守れる |

**AI時代との接続：**
AIとの協働は本質的に「非同期」だ。AIにタスクを依頼し、結果を確認し、フィードバックする——このサイクルは、同期コミュニケーション前提の組織より、非同期文化の組織の方がはるかに自然に実装できる。

---

### 原則3：シングル・ソース・オブ・トゥルース（SSOT）

**「情報の正本は1箇所にしか存在しない」**

情報が散在する組織では、「どれが最新か」「どれが正しいか」の確認に時間が消える。
GitLabはすべての情報の「正本」を1つのハンドブックに集約し、**更新されたら全員が同じ情報を参照できる**状態を維持している。

```
情報が散在している組織
  メール / Slack / 共有ドライブ / 個人PCに分散
  → 「最新版はどこ？」の確認が発生
  → 認識のズレで手戻りが起きる
  → AIに渡しても「どれが正しいか」分からない

SSOT（単一ソース）がある組織
  → 全員が同じ情報を参照している
  → 確認コストがゼロに近い
  → AIにそのまま渡せる
```

---

### 原則4：透明性のデフォルト化

**「隠す理由がないかぎり、公開する」**

多くの組織では「必要な人だけに共有」が默黙のルールになっている。これは情報の流通を妨げ、サイロ化を生む。

GitLabは「デフォルト・パブリック」の文化を持つ。社内情報の大半が全社員に公開され、必要なときに必要な情報へアクセスできる。

**組織変革との接続：**
変革が失敗する大きな原因の一つは「情報格差」だ。経営が見ている絵と現場が見ている絵が違うとき、変革は形骸化する。透明性の設計は、コッターの変革8段階でいう「緊急性の醸成」と「変革ビジョンの共有」を構造的に支援する。

> **参考フレーム：** [コッターの変革8段階](/docs/フレームワーク・参考書/フレームワーク/コッターの変革8段階) の第1〜3段階（緊急性、連帯、ビジョン）は、透明性の構造なしには機能しにくい。

---

### 原則5：成果ベース評価（アウトカム・ドリブン）

**「何時間働いたか」ではなく「何を達成したか」で評価する**

リモートの文脈で語られることが多いが、これは**AI時代の評価原則**でもある。

AIが「時間を使う仕事」を代替していく中で、人間の価値は「何を判断し、何を実現したか」に移行する。評価制度が稼働時間・作業量ベースのままでは、AI活用を促進するインセンティブが生まれない。

> **参考フレーム：** [OKR](/docs/フレームワーク・参考書/フレームワーク/OKR) はこのアウトカム・ドリブンを組織的に実装するための仕組み。「何に挑戦したか」「何を達成したか」を評価対象にすることで、AI活用・変革参加が正しく報われる評価構造を作る。

## 「リモート組織」はなぜ「AIレディ組織」なのか

```
リモート組織の必須要件          AI活用の必須要件
─────────────────────────────────────────────────
ドキュメントに書く          →  AIに渡せる情報がある
非同期で動く               →  AIとのサイクルに適応できる
SSOTで管理する             →  AIに正確なコンテキストを渡せる
透明性が高い               →  AIが参照できる社内知識が多い
成果ベースで評価する        →  AI活用を妨げるインセンティブがない
─────────────────────────────────────────────────
```

リモート組織を成功させるために解いてきた課題が、そのままAI組織化の課題と重なる。
GitLabが「AIレディ組織の教科書」として読める理由がここにある。

## 日本企業が詰まりやすいポイント

### ① 「ドキュメントを書く時間がない」問題

現場は既存業務で手一杯で、ドキュメント化が後回しになる悪循環。
打開策は「ドキュメント化を別タスクとして追加しない」こと。
会議の議事録をAIに生成させる、Slackの議論をAIで要約してドキュメントに落とすなど、**AIをドキュメント化のコストを下げるツールとして使う**のが現実的な第一歩。

### ② 「全部書くのは無理」という思い込み

完璧なドキュメントを作ろうとすると始められない。
GitLabの哲学は「完璧より公開」——70%の状態でも公開して、後から育てる。
イテレーティブな改善がドキュメントにも適用される。

> **参考フレーム：** [アジャイル・スクラム](/docs/フレームワーク・参考書/フレームワーク/アジャイル・スクラム) の「動くものを最速で作る」発想が、ドキュメント化にも適用できる。

### ③ 透明性への文化的抵抗

「情報は権力」という暗黙の組織文化では、透明性のデフォルト化は強い抵抗に遭う。
これは技術の問題ではなく、[心理的安全性](/docs/フレームワーク・参考書/フレームワーク/心理的安全性) の問題として扱う必要がある。

---

## こんな課題に効く

- 業務の属人化が激しく、担当が変わると品務が止まる（B-2）
- やるべきことは分かっているが日常業務が重く変革に着手できない（C-6）
- 部門をまたいだプロジェクトで情報が共有されず重複作業が起きる（C-4）
- AIを導入したが組織全体には広がらない（G-2）
- 変革を推進する人が孤立する。「うちには無理」という声が出る（B-4）
