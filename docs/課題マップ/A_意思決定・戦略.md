## このカテゴリの症状

「決まらない」「動けない」「方向性がブレる」という痛みを抱えている組織へ。

---

### A-1. 意思決定が遅い・止まる

**症状**
- 会議をしても何も決まらない
- 承認フローが長すぎて施策が動き出さない
- 誰が最終意思決定者か分からない
- リスクを取ることを誰も決断しない

**真因の仮説**

**① 意思決定権限が階層の上に集中し、現場の判断余地がない**（妥当性：★★★ 高）
承認フローが長いのは、組織が「誰かが失敗したとき責任を取れる人間を明確にする」ために権限を上位に集中させた結果です。その構造の中では、現場は「上が決めるまで動かない」ことが合理的な行動になります。承認を求める側も、承認する側も、リスクを回避しようとするため、決定が遅くなるのは構造の必然です。
> 📚 [HIGH OUTPUT MANAGEMENT](/docs/フレームワーク・参考書/書籍要約/HIGH-OUTPUT-MANAGEMENT) — アンドリュー・グローブは「意思決定は最も情報を持つ者が行う」原則を説く。承認階層は情報の流れに逆行しており、管理職が権限委譲をしないかぎり意思決定は遅くなる必然があります。

**② 会議の場で初めてデータが共有され、観察フェーズと判断フェーズが分離されていない**（妥当性：★★★ 高）
日本の多くの組織では、会議が「情報共有の場」と「意思決定の場」を兼ねています。参加者が初めて情報に触れる場で判断を求められるため、全員が「もう少し情報が欲しい」と感じ、決定が先送りされます。
> 📚 [OODA vs PDCA](/docs/フレームワーク・参考書/フレームワーク/OODA-vs-PDCA) — OODAループは「観察（Observe）→方向付け（Orient）→決定（Decide）→行動（Act）」を分けて設計します。会議の前に観察・方向付けが完了していないと、判断ステップに到達できません。

**③ 心理的安全性が低く、少数意見や懸念が表明されないまま「なんとなく合意」で終わる**（妥当性：★★☆ 中）
反対意見を言うコストが高い組織では、表面上は合意に見えても「誰も本気で賛成していない決定」が生まれます。実行段階になって「やはり難しい」という声が出始め、プロジェクトが止まります。
> 📚 [心理的安全性](/docs/フレームワーク・参考書/フレームワーク/心理的安全性) — エイミー・エドモンドソンの研究では、心理的安全性が低いチームほど「発言コストの高い情報」が隠蔽され、意思決定の質が下がることが示されています。

**関連する変革手段**
- [知見・意思決定の変革](/docs/変革マップ/知見・意思決定の変革/概要) — 外部知見で判断材料を補う
- [実行・プロセスの変革](/docs/変革マップ/実行・プロセスの変革/概要) — 意思決定プロセス自体を見直す
- [体制・人材の変革](/docs/変革マップ/体制・人材の変革/概要) — 意思決定できる人材を入れる

### A-2. 何をすべきか判断できない（情報不足）

**症状**
- 市場の動きが読めず、方針が立てられない
- 自社の強み・弱みを客観視できていない
- 競合が何をしているか分からない
- 顧客が本当に何を求めているか分からない

**真因の仮説**

**① 業界の第一線にいる人材が社内におらず、「生きた情報」が入ってこない**（妥当性：★★★ 高）
経営者や幹部が特定の業界・市場に長くいると、自社の常識が業界の常識と一致しているように見えます。しかし実際には変化が起きており、その変化に気づくためのアンテナ役が社内にいないため、外から見れば明らかな変化が内側からは見えないという構造が生まれます。
> 📚 [スポットコンサル・インタビュー](/docs/変革マップ/知見・意思決定の変革/スポットコンサル・インタビュー) — 業界に精通した外部専門家との対話が、社内の認識バイアスを壊す最も効果的な手段の一つです。

**② 情報収集が個人の努力に依存しており、組織的な仕組みとして機能していない**（妥当性：★★☆ 中）
優秀な個人が勉強会やカンファレンスから情報を持ち帰っても、その情報が組織の意思決定に接続される仕組みがなければ、個人の知識で終わります。情報が「組織の判断材料」になるための収集・整理・共有プロセスが設計されていないことが根本原因です。
> 📚 [ダブルダイアモンド](/docs/フレームワーク・参考書/フレームワーク/ダブルダイアモンド) — 課題定義フェーズの「発散」では、複数の情報源から広く情報を集め、フレームを疑うことが求められます。個人頼みの情報収集では発散が起きず、正しい課題設定ができません。

**関連する変革手段**
- [スポットコンサル・インタビュー](/docs/変革マップ/知見・意思決定の変革/スポットコンサル・インタビュー) — 専門家から直接知見を得る
- [市場調査・リサーチ](/docs/変革マップ/知見・意思決定の変革/市場調査・リサーチ) — 市場・競合・顧客を構造的に把握する

### A-3. 経営と現場の乖離

**症状**
- 経営が決めた方針が現場に届かない
- 現場の課題・声が経営に上がらない
- 部門によって動く方向が違う
- 「言ったことをやってくれない」と経営が感じている

**真因の仮説**

**① 経営方針が「何のために・何を・どこまで」という目的と基準に翻訳されないまま現場に降りている**（妥当性：★★★ 高）
「イノベーションを推進する」「顧客第一で動く」という方針は、現場が行動を変えるための具体的な基準を持ちません。現場は既存の業務慣行に従って動くのが合理的であり、方針が現場行動に変換されないのは意図的な抵抗ではなく、翻訳の仕組みが欠けているためです。
> 📚 [OKR](/docs/フレームワーク・参考書/フレームワーク/OKR) — OKRは経営方針を「Objective（定性目標）× Key Results（定量指標）」の形式で現場レベルまで分解することで、方針と現場行動のギャップを埋める仕組みです。目標の具体化なしに整合はとれません。

**② 縦割り組織構造が情報フローを遮断し、経営と現場の「共通の現実」を作れない**（妥当性：★★☆ 中）
部門が縦割りになっていると、各部門長が経営からの情報を独自に解釈・フィルタリングして下に伝えます。その結果、経営が見ている現実と現場が見ている現実が異なり、「言ったはずのことがやられていない」という状況が生まれます。
> 📚 [チームトポロジー](/docs/フレームワーク・参考書/書籍要約/チームトポロジー) — チーム間のインタラクションモードを設計しないと、縦割りサイロが自然に生まれます。経営と現場の情報フローは意図的に設計されなければなりません。

**③ 目標・評価制度が部門最適を促進し、全社方針と個人行動を乖離させている**（妥当性：★★☆ 中）
部門KPIが全社方針と整合していない場合、社員は評価されるKPIに従って動きます。経営が「横断連携」を求めても、評価が「部門の数字」でなされる限り、社員は合理的に部門内に閉じて行動します。
> 📚 [BizOps・RevOps](/docs/フレームワーク・参考書/フレームワーク/BizOps-RevOps) — 収益プロセスをまたいだKPI統合がBizOpsの核心です。部門ごとに最適化されたKPIが全体最適を阻害する構造は、あらゆる組織で観察されます。

**関連する変革手段**
- [実行支援・PMO](/docs/変革マップ/実行・プロセスの変革/実行支援・PMO) — 経営と現場をつなぐ役割を外部で担う
- [顧問・アドバイザリー](/docs/変革マップ/体制・人材の変革/顧問・アドバイザリー) — 組織設計・マネジメントの専門家を入れる

### A-4. 新規事業・新しい挑戦が生まれない

**症状**
- アイデアは出るが承認されない
- 承認されても実行できない
- そもそもアイデアが出てこない
- 新規事業を始めたが育たずに終わる

**真因の仮説**

**① 既存事業の評価基準（確実性・収益性・ROI）で新規事業を判断するため、初期段階のアイデアが全て却下される**（妥当性：★★★ 高）
新規事業は本質的に不確実性が高く、ROIを事前に計算できません。しかし多くの組織では、既存事業への投資判断と同じ基準で新規事業を評価するため、「まだ売上が見えない段階」のアイデアは全て承認されません。これは制度的な問題であり、個人のリスク感覚の問題ではありません。
> 📚 [両利きの経営](/docs/フレームワーク・参考書/書籍要約/両利きの経営) — 「探索（Exploration）」と「深化（Exploitation）」は異なる評価基準・組織構造・文化を必要とします。既存事業の論理で新規事業を管理することは、探索を構造的に殺す行為です。

**② 失敗が「人の問題」として処理され、仮説検証の文化が育たない**（妥当性：★★★ 高）
新規事業の失敗を「担当者の判断ミス」として評価する組織では、誰も新しい挑戦のオーナーになりません。失敗は仮説検証プロセスの一部であるという認識が共有されていないと、チャレンジそのものが消滅します。
> 📚 [リーンスタートアップ](/docs/フレームワーク・参考書/書籍要約/リーンスタートアップ) — MVPによる高速な仮説検証は「失敗を早く・安く・学びに変える」設計思想です。失敗を責める文化ではこのサイクルが機能せず、新規事業の探索が止まります。

**③ 新規事業に必要なスキル（顧客開発・プロダクト設計・資金調達）を持つ人材が社内にいない**（妥当性：★★☆ 中）
既存事業のオペレーションに最適化された人材は、不確実性の高い環境での意思決定や顧客との対話から仮説を作るスキルを持っていないことが多いです。スキルではなく意欲の問題として片付けられがちですが、実態は能力ギャップです。
> 📚 [越境学習・経験付与](/docs/変革マップ/人・能力の変革/越境学習・経験付与) — 新規事業に必要なスキルは既存事業の中では育ちません。外部環境への越境が、異なる思考回路の獲得を促します。

**関連する変革手段**
- [越境学習・経験付与](/docs/変革マップ/人・能力の変革/越境学習・経験付与) — 外部環境に触れることでマインドセットを変える
- [外部人材活用](/docs/変革マップ/体制・人材の変革/外部人材活用) — 新規事業経験を持つ人材を外部から入れる
- [戦略的アライアンス](/docs/変革マップ/販路・外部資本の変革/戦略的アライアンス) — 外部との連携で新領域に踏み出す

### A-5. 変化への対応が後手に回る

**症状**
- 業界の変化に気づいたときにはもう遅い
- 競合の新しい動きに反応できない
- デジタル化・AI活用で他社に遅れをとっている
- 外部環境の変化を社内で共有する仕組みがない

**真因の仮説**

**① 環境変化を観察する役割・仕組みが制度化されておらず、情報が「偶然知る」ものになっている**（妥当性：★★★ 高）
業界変化への対応が遅い組織の多くは、変化を監視する専任の役割や仕組みを持ちません。個人が偶然カンファレンスや記事で気づいても、その情報が経営の判断に接続されるプロセスがなければ、実質的に情報として機能しません。
> 📚 [OODA vs PDCA](/docs/フレームワーク・参考書/フレームワーク/OODA-vs-PDCA) — OODAループの「観察（Observe）」は継続的・組織的な情報収集を前提とします。観察が偶発的なPDCAサイクルでは、変化への反応が常に後手になります。

**② 情報を受け取っても意思決定に変換する仕組みがないため、「知っているが動けない」状態が続く**（妥当性：★★☆ 中）
DXや競合動向についての情報は既に社内に存在していても、「誰が・いつまでに・何を決めるか」のプロセスがなければ情報は行動に変わりません。意思決定の遅さ（A-1）と連動して、情報があっても動けない構造が形成されます。
> 📚 [Society 5.0](/docs/フレームワーク・参考書/フレームワーク/Society5.0) — Society 5.0が示す環境変化のスピードは、PDCAのような長い計画サイクルでは対応できません。リアルタイムに近い情報と判断のループが必要です。

**関連する変革手段**
- [スポットコンサル・インタビュー](/docs/変革マップ/知見・意思決定の変革/スポットコンサル・インタビュー)
- [市場調査・リサーチ](/docs/変革マップ/知見・意思決定の変革/市場調査・リサーチ)
- [越境学習・経験付与](/docs/変革マップ/人・能力の変革/越境学習・経験付与)

### A-6. 課題・問題が言語化されていない／暗黙知として放置されている

**症状**
- 「なんかうまくいっていない」という感覚はあるが、何が問題か言えない
- 課題が人によってバラバラで、組織として認識が揃っていない
- ベテランのノウハウや判断基準が頭の中だけにあり、共有されていない
- 問題提起をしても「それは仕方ない」で片付けられる
- 振り返りや内省の文化・習慣がない

**真因の仮説**

**① 日常業務の処理に追われ、立ち止まって構造的に考える時間と場が設計されていない**（妥当性：★★★ 高）
課題が言語化されない最大の理由は、「言語化する余白がない」ことです。月次会議が報告の場として機能しているが内省の場として機能していない組織では、問題を「感じている」ことと「語れる」ことの間に大きなギャップが生まれます。問題が見えていても、それを言葉にする時間・場・構造がなければ暗黙知として残り続けます。
> 📚 [アジャイル・スクラム](/docs/フレームワーク・参考書/フレームワーク/アジャイル・スクラム) — スクラムのレトロスペクティブ（振り返り）は「何がうまくいったか・いかなかったか」を定期的に構造化する場です。この場がなければ問題は感覚のまま残ります。

**② 問題を言語化することが「批判・不満」として受け取られる文化があり、言語化自体が抑制される**（妥当性：★★★ 高）
「問題を言うと面倒なことになる」という学習が組織に蓄積すると、問題の感知そのものが抑制されるようになります。これは個人の消極性ではなく、発言コストが高すぎる組織設計の問題です。
> 📚 [心理的安全性](/docs/フレームワーク・参考書/フレームワーク/心理的安全性) — 心理的安全性が低い組織では、問題の言語化は「リスクある行動」として認識されます。安全性の確保なしに、課題の言語化を促すことはできません。

**③ 問題を構造化するためのフレームワーク・語彙が共有されておらず、議論がふわっとしたまま終わる**（妥当性：★★☆ 中）
「何かおかしい」という感覚はあっても、それを「誰が・何を・なぜ・どのくらい」という構造に分解するスキルがなければ、議論は「そうだよね」「難しいよね」で終わります。問題の言語化は技術であり、トレーニングと外部の視点なしに自然発生しません。
> 📚 [ダブルダイアモンド](/docs/フレームワーク・参考書/フレームワーク/ダブルダイアモンド) — 課題定義（Problem Framing）は意図的に設計された発散・収束プロセスを必要とします。フレームなしに「問題を語る会議」を開いても、課題は収束しません。

**関連する変革手段**
- [スポットコンサル・インタビュー](/docs/変革マップ/知見・意思決定の変革/スポットコンサル・インタビュー) — 外部の専門家との対話が言語化の契機になる
- [市場調査・リサーチ](/docs/変革マップ/知見・意思決定の変革/市場調査・リサーチ) — 外部視点と照らすことで自社課題が浮かび上がる
- [越境学習・経験付与](/docs/変革マップ/人・能力の変革/越境学習・経験付与) — 外の世界を見ることで「当たり前」が相対化される
- [顧問・アドバイザリー](/docs/変革マップ/体制・人材の変革/顧問・アドバイザリー) — 外部の目で課題を構造化してもらう
- [実行支援・PMO](/docs/変革マップ/実行・プロセスの変革/実行支援・PMO) — 課題整理・論点整理を伴走してもらう
